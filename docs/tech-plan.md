# 时间管理 Web 项目技术方案

本文档基于当前 [product-spec.md](/D:/JH/docs/product-spec.md) 进行细化，目标是为第一版 MVP 提供一套 **简单、稳定、容易维护、适合零基础开发者上手** 的技术实现方案。

> 补充说明：根据当前需求，本技术方案将 **任务编辑、任务删除** 也纳入第一版 MVP，方便第一版直接形成完整的任务闭环。

## 1. 技术栈选择

### 1.1 前端框架

推荐选择：**React + Vite + React Router**

说明：

- `React`：适合做页面拆分和组件复用
- `Vite`：启动快、配置少、适合快速搭建项目
- `React Router`：负责页面切换，满足 dashboard、今日视图、本周视图、番茄钟页、统计页的路由需求

### 1.2 语言

推荐选择：**TypeScript**

说明：

- 第一版只使用最基础的类型定义，不使用复杂语法
- 对零基础开发者来说，TypeScript 虽然比 JavaScript 多一点语法，但能更早发现字段写错、数据为空、类型不一致等问题
- 后续继续扩展任务编辑、统计、番茄钟状态时，类型约束会更稳定

建议控制原则：

- 只用 `type` 或 `interface` 定义数据结构
- 不使用复杂泛型技巧
- 不使用 class
- 以“能看懂、能改动”为优先

### 1.3 样式方案

推荐选择：**普通 CSS**

说明：

- 使用成本最低
- 不需要理解额外框架语法
- 出问题时最容易直接定位
- 方便零基础开发者修改页面样式

建议做法：

- 保留一个全局样式文件：`src/styles/global.css`
- 保留一个变量文件：`src/styles/variables.css`
- 每个页面或组件可以有独立 CSS 文件，例如 `TaskCard.css`

第一版不建议使用：

- Tailwind CSS
- Sass / Less
- CSS-in-JS

这些方案并不是不能用，而是会增加第一版理解成本。

### 1.4 数据存储方案

推荐选择：**浏览器 localStorage**

说明：

- 不需要后端
- 不需要数据库
- 不需要登录
- 刷新页面后数据仍可保留
- 非常适合 MVP 阶段快速验证产品是否能用

限制也要提前明确：

- 数据只保存在当前浏览器
- 换设备或清空浏览器缓存后会丢失
- 不适合多端同步

### 1.5 日期处理方案

推荐选择：**dayjs**

说明：

- 体积小
- API 简单，比原生 `Date` 更容易理解
- 适合做“今天”“本周”“截止时间格式化”“是否逾期”等常见逻辑

建议统一规则：

- 存储时使用 ISO 字符串，例如 `2026-04-10T20:30:00.000Z`
- 展示时使用本地时间格式化
- 今日 / 本周 / 逾期状态都通过函数动态计算

### 1.6 为什么这样选

第一版的核心目标不是“技术先进”，而是“尽快做出可用版本并且后面容易维护”。因此这一套组合有几个明显优势：

- 学习成本低
- 项目依赖少
- 页面和组件结构清楚
- 很适合后续继续让 AI 直接生成代码
- 不需要后端，能把注意力集中在业务功能上

总结为一句话：

**React 负责页面，TypeScript 保证基本稳定，普通 CSS 控制复杂度，localStorage 解决数据保存，dayjs 解决日期问题。**

## 2. 项目目录结构

推荐目录结构如下：

```text
src/
  assets/
  components/
    common/
      AppLayout.tsx
      PageHeader.tsx
      EmptyState.tsx
    task/
      TaskForm.tsx
      TaskList.tsx
      TaskCard.tsx
      TaskFilters.tsx
    pomodoro/
      PomodoroTimer.tsx
      TimerControls.tsx
    stats/
      StatCard.tsx
  pages/
    DashboardPage.tsx
    TodayPage.tsx
    WeekPage.tsx
    PomodoroPage.tsx
    StatsPage.tsx
  hooks/
    useTasks.ts
    usePomodoro.ts
  utils/
    date.ts
    storage.ts
    task.ts
    stats.ts
  types/
    task.ts
    pomodoro.ts
  styles/
    variables.css
    global.css
  router/
    index.tsx
  App.tsx
  main.tsx
```

目录说明：

- `pages/`：放页面组件，一个文件对应一个页面
- `components/`：放可复用组件
- `hooks/`：放业务 Hook，例如任务管理、番茄钟逻辑
- `utils/`：放纯函数，例如日期判断、统计计算、localStorage 读写
- `types/`：放数据结构定义
- `styles/`：放全局样式和基础变量
- `router/`：集中管理路由配置

第一版不要一开始就加这些目录：

- `services/`
- `api/`
- `store/`
- `constants/` 过多拆分
- `tests/` 复杂测试工程

不是不能加，而是 MVP 阶段容易把结构拆得过细，反而让零基础开发者更难理解。

## 3. 页面拆分

### 3.1 `dashboard`

负责内容：

- 显示今日待办数量
- 显示今日已完成数量
- 显示本周待办概览
- 显示当前番茄钟状态
- 提供“新增任务”的快捷入口
- 提供跳转到今日视图、本周视图、番茄钟页、统计页的入口

页面定位：

- 作为项目首页
- 让用户一进来就知道“今天有什么事”和“现在该做什么”

### 3.2 今日视图

负责内容：

- 展示今天需要处理的任务
- 支持新增任务
- 支持编辑任务
- 支持删除任务
- 支持切换完成状态
- 支持按截止时间查看任务

建议展示规则：

- 先显示未完成任务
- 再显示已完成任务
- 可以显示已逾期但未完成的任务

### 3.3 本周视图

负责内容：

- 展示本周 7 天内的任务
- 按日期分组显示
- 支持任务完成状态切换
- 支持编辑和删除任务

页面定位：

- 帮助用户从“今天”扩展到“这周”
- 用来查看任务整体分布，而不是只看单日

### 3.4 番茄钟页

负责内容：

- 显示当前倒计时
- 支持开始、暂停、重置
- 显示当前状态，例如“未开始 / 进行中 / 已暂停 / 已结束”
- 可选关联当前专注任务，但第一版不强制绑定任务

页面定位：

- 独立承载专注功能
- 避免和任务页面混在一起导致页面太乱

### 3.5 统计页

负责内容：

- 显示总任务数
- 显示已完成任务数
- 显示未完成任务数
- 显示今日完成数
- 显示本周完成数
- 显示完成率

页面定位：

- 用于展示结果
- 第一版只显示数字统计，不做复杂图表

## 4. 组件拆分

第一版建议保留以下核心组件。

### 4.1 `AppLayout`

作用：

- 提供统一页面布局
- 放导航栏和主要内容区域

### 4.2 `PageHeader`

作用：

- 显示页面标题和简单说明
- 保持页面头部风格统一

### 4.3 `TaskForm`

作用：

- 用于新增任务
- 也用于编辑任务

建议字段：

- 任务标题
- 截止时间

说明：

- 第一版可以使用“新增 / 编辑共用一个表单”
- 通过是否传入已有任务数据来区分模式

### 4.4 `TaskList`

作用：

- 负责接收任务数组并循环渲染
- 统一处理空状态展示

### 4.5 `TaskCard`

作用：

- 展示单条任务信息
- 提供完成、编辑、删除操作按钮
- 显示截止时间和逾期状态

### 4.6 `TaskFilters`

作用：

- 负责简单筛选，例如全部 / 未完成 / 已完成

说明：

- 第一版如果开发时间紧，也可以先不做成独立组件

### 4.7 `EmptyState`

作用：

### 4.8 `PomodoroTimer`

作用：

- 显示剩余时间
- 负责倒计时逻辑的界面展示

### 4.9 `TimerControls`

作用：

- 提供开始、暂停、重置按钮

### 4.10 `StatCard`

作用：

- 用于展示统计数字卡片
- 避免统计页写一堆重复结构

## 5. 数据模型

### 5.1 任务数据结构

推荐任务结构如下：

```ts
export interface TaskItem {
  id: string;
  title: string;
  deadline: string | null;
  completed: boolean;
  createdAt: string;
  completedAt: string | null;
}
```

字段说明：

- `id`：任务唯一标识，建议使用时间戳加随机字符串
- `title`：任务标题
- `deadline`：截止时间，未设置时可为 `null`
- `completed`：是否完成
- `createdAt`：创建时间
- `completedAt`：完成时间，未完成时为 `null`

### 5.2 `description` 是否需要

建议：**第一版不需要**

原因：

- 会增加表单复杂度
- 会增加列表展示复杂度
- MVP 目标是先把任务闭环做通

如果想为后续扩展做准备，可以在后面版本再增加：

```ts
description?: string;
```

但第一版不建议先加进表单和页面。

### 5.3 `priority` 是否第一版需要

建议：**第一版不需要**

原因：

- 会增加筛选和排序规则
- 会让用户操作变复杂
- 对 MVP 的核心闭环不是必须项

第一版先保证：

- 能新增
- 能编辑
- 能删除
- 能完成
- 能按今日 / 本周查看

等这些稳定后，再考虑优先级。

### 5.4 `overdue` 状态是否需要单独存

建议：**不要单独存**

原因：

- `overdue` 是一个“计算结果”，不是原始数据
- 它可以通过 `deadline + completed + 当前时间` 动态得出
- 如果单独保存，容易出现数据不同步问题

推荐写成工具函数：

```ts
isTaskOverdue(task, now)
```

这样最容易维护。

## 6. 状态管理方案

推荐方案：**React 自带状态 + 自定义 Hook**

建议做法：

- 任务数据由 `useTasks` 管理
- 番茄钟状态由 `usePomodoro` 管理
- 页面通过 Hook 读取和操作数据

例如：

- `useTasks` 负责新增、编辑、删除、切换完成、保存到 localStorage
- `usePomodoro` 负责开始、暂停、重置、倒计时恢复

为什么这样选：

- 不需要 Redux
- 不需要 MobX
- 不需要 Zustand
- 对零基础开发者更友好
- 文件职责清楚

第一版不建议做：

- 全局复杂状态树
- `useReducer` 大量 action
- 多层 Context 嵌套

最简单可维护的思路是：

**把任务逻辑放进一个 Hook，把番茄钟逻辑放进一个 Hook，再把页面当成展示层。**

如果后面页面之间共享数据变多，再考虑增加一个简单的 `AppContext`，但第一版可以先不做过度设计。

## 7. 本地存储方案

### 7.1 localStorage Key 设计

建议使用固定 key：

- `time-manager.tasks`
- `time-manager.pomodoro`

这样后续维护时一眼就能知道每份数据保存在哪里。

### 7.2 任务保存方案

任务数据建议直接保存整个数组：

```ts
[
  {
    "id": "task_1712730000000_ab12",
    "title": "完成周报",
    "deadline": "2026-04-12T10:00:00.000Z",
    "completed": false,
    "createdAt": "2026-04-10T08:00:00.000Z",
    "completedAt": null
  }
]
```

建议流程：

1. 页面初始化时从 localStorage 读取任务数组
2. 如果没有数据，返回空数组
3. 每次新增、编辑、删除、切换完成后，立即重新保存

建议封装函数：

- `loadTasks()`
- `saveTasks(tasks)`

### 7.3 番茄钟保存方案

番茄钟建议保存为一个对象：

```ts
export interface PomodoroState {
  durationSeconds: number;
  remainingSeconds: number;
  status: "idle" | "running" | "paused" | "finished";
  endTime: string | null;
}
```

字段说明：

- `durationSeconds`：总时长，第一版固定 1500 秒
- `remainingSeconds`：剩余秒数
- `status`：当前状态
- `endTime`：如果正在运行，记录预计结束时间

为什么要存 `endTime`：

- 如果用户刷新页面，可以根据当前时间重新计算剩余秒数
- 这样比只存 `remainingSeconds` 更稳定

建议流程：

1. 开始时写入 `status=running` 和 `endTime`
2. 暂停时更新 `remainingSeconds`，并把 `endTime` 设为 `null`
3. 重置时恢复默认值
4. 页面加载时如果发现状态是 `running`，则根据 `endTime` 重新计算剩余时间

### 7.4 localStorage 使用注意

- 读写前都要加 `try/catch`
- 解析 JSON 失败时要回退默认值
- 不要把函数存进 localStorage
- 只存纯数据对象

## 9. 风险提醒

第一版最容易出错的地方主要有以下几类。

### 9.1 日期判断容易出错

风险点：

- “今天”与“本周”的判断规则不统一
- ISO 时间和本地展示时间混淆
- 任务在临界时间附近显示错误

建议：

- 所有日期处理统一走 `dayjs`
- 统一在 `utils/date.ts` 中处理，不要把日期逻辑散落到各页面

### 9.2 localStorage 数据不同步

风险点：

- 新增后没有及时保存
- 编辑后只更新页面，没有更新 localStorage
- 删除后刷新又出现旧数据

建议：

- 所有任务变更都只通过 `useTasks` 处理
- 不要在页面里直接写 localStorage

### 9.3 逾期状态存错

风险点：

- 如果把 `overdue` 直接存起来，当前时间变化后就可能不准确

建议：

- 逾期状态一律动态计算，不要入库，不要入 localStorage

### 9.4 番茄钟刷新后状态不准

风险点：

- 只保存剩余秒数，刷新后会丢失真实进度
- 浏览器标签页切换后计时不稳定

建议：

- 运行中一定保存 `endTime`
- 页面恢复时用“当前时间和结束时间差值”重新计算

### 9.5 编辑和删除影响统计

风险点：

- 删除已完成任务后，统计值可能突然变化
- 编辑截止时间后，今日 / 本周统计可能变化

建议：

- 接受统计会实时变化这一事实
- 所有统计都基于“当前任务列表”即时计算，不额外缓存

### 9.6 页面职责不清会越写越乱

风险点：

- 页面里直接写太多业务逻辑
- 同样的任务卡片在多个页面重复写

建议：

- 页面负责拼装数据和布局
- 组件负责展示
- Hook 负责业务逻辑
- utils 负责纯计算

## 10. 结论

第一版最适合的路线是：

- 使用 `React + Vite + TypeScript`
- 使用普通 CSS
- 使用 `dayjs` 处理日期
- 使用 `localStorage` 保存任务和番茄钟数据
- 使用 `useTasks` 和 `usePomodoro` 这两个核心 Hook 管理业务

这套方案最大的优点是：

- 上手快
- 出错点少
- 后续容易扩展
- 很适合继续直接生成代码

后续如果按这份方案继续实现，建议从“项目初始化 + 路由 + 任务数据模型”开始，一步一步往下做，不要一开始就把所有功能一起堆上去。
